# 2026-06-22 — Multi-node Raft over the fabric + split-brain mitigation

Made the Raft control plane **real across hosts** and closed the split-brain
gaps: the daemon now runs consensus over the encrypted fabric, forms clusters
from seeds, and the failure detector is quorum-gated so a partition can't
double-schedule.

| Area | Change |
|------|--------|
| Cross-host consensus | Daemon runs Raft over the fabric transport (`start_fabric` + `serve_raft`) instead of an in-process router |
| Cluster formation | No seeds → quorum-of-one; seeds → join an existing cluster (leader `add_learner` + `change_membership`) |
| Split-brain | Quorum on writes (minority can't commit) + **quorum-gated** failure detector (only the leader reschedules) |

Detail: [ocf-consensus → Cluster formation & split-brain](../subsystems/ocf-consensus.md#cluster-formation--split-brain).

---

## 1. Raft over the encrypted fabric (cross-host)

Previously each daemon node ran its own **in-process**, single-voter Raft (a
quorum of one) — there was no cross-node consensus. Now:

- `ReplicatedStore::start_fabric(node_id, store, transport, server)` builds the
  Raft node with the `FabricRaftNetworkFactory` (RPCs over the node's Noise
  identity) and spawns `serve_raft` on a bound `FabricServer`. The daemon binds it
  on `fabric_raft_port` (51901). Raft rides the `wg-mgmt` plane, so a NAT'd node
  can participate.
- New `ReplicatedStore` cluster ops: `initialize_cluster(members+addrs)`,
  `add_learner(id, addr)`, `change_membership(voters)`, `voters()`.

## 2. Cluster formation from seeds

- **No seeds** (demo / first node): `initialize_cluster([self])` — a quorum of one.
  Every write is still ordered through the Raft log. Boots in ~1 s.
- **With seeds** (joining node): starts **empty** (a learner has no leader to
  write through, so it must not seed), then `join_cluster` advertises its Raft
  endpoint to each seed's control channel via a `join <json>` verb. The seed, *if
  it is the leader*, runs `add_learner` then `change_membership` to promote it;
  otherwise it replies `notleader` and the joiner tries another seed. State
  replicates in from the leader.

## 3. Split-brain mitigation

- **Quorum on every write.** A command commits only with a majority, so a minority
  partition can't commit — it has no leader. Proven by new tests:
  `fabric_cluster_forms_and_replicates`, `follower_write_is_redirected_…` (no
  second writer), and `minority_partition_cannot_commit` (a partitioned minority is
  inert) — all through the same `start_fabric` path the daemon uses.
- **Quorum-gated failure detector.** `on_node_dead` reschedules HA workloads only
  when `consensus.is_leader()`. In a partition both detectors fire, but only the
  majority side has a leader, so exactly one node reschedules — two halves can't
  both restart the same HA workload. (A quorum-of-one node is always its own
  leader, so the single-node path is unaffected.)

## 4. Package updates, security warnings & OSV vulnerability scanning

A package-update handler, a security-update warning, and OSV-based vulnerability
scanning of installed packages — built on `ocf-platform` + `ocf-health`.

- **Update handler.** The `PackageManager` contract gains `list_updates`,
  `apply_updates(security_only)`, and `list_installed_packages`, implemented for
  apt/dnf/pacman/apk. apt and dnf flag **security** updates (apt by the
  `…-security` suite, dnf via `updateinfo list security`) and support applying
  security-only (`apt-get install --only-upgrade …` / `dnf upgrade --security`).
  Output parsing lives in `update.rs` as pure, unit-tested functions.
  `PlatformService` exposes `available_updates()`, `apply_updates()`,
  `scan_vulnerabilities()`. New endpoints `GET /api/v1/platform/updates`,
  `POST /api/v1/platform/updates/apply`, `GET /api/v1/platform/vulnerabilities`.
- **Security-update warning.** A `SecurityUpdateCheck` (health) raises a `Warning`
  when security updates are pending, with one-press *Apply security updates* /
  *Apply all updates* fixes — surfaced at `GET /api/v1/health/findings`.
- **OSV scanning.** `OsvClient` queries the public OSV batch API
  (`api.osv.dev/v1/querybatch`) with installed packages mapped to their distro
  ecosystem (Ubuntu/Debian/Red Hat/Alpine), chunked to OSV's batch limit; a
  `VulnerabilityCheck` raises a `Critical` finding listing matched packages. The
  request build / response parse are pure-tested, and a `#[ignore]`d live test
  confirms a real query to api.osv.dev reports a known CVE. Added `ureq` (rustls)
  and a `Security` health category.

## 5. LAN deployment: auto-detect the node's reachable address

So a fleet on a local network behind a router (no public IP) works without
hardcoding addresses:

- `ocf_fabric::detect_local_address()` finds the host's primary IPv4 toward the
  default route (the **router-assigned LAN address** behind NAT), correct even
  with multiple NICs and sending no packets (a connected UDP socket only selects
  a route). On boot the controller resolves this node's address — an explicit
  `--fabric-address` / `OCF_FABRIC_ADDRESS` override, else auto-detect — and
  applies it to this node's own machine, so the mesh, WireGuard, and Raft
  advertise the real address rather than a seeded placeholder. Verified live: a
  node picked up its host's actual `10.0.1.x` LAN IP, and the override won when
  set.

## 6. Automatic node identity (no manual node id)

The operator no longer has to name a node. `--node-id`/`OCF_NODE_ID` is now
**optional**; when omitted, `ocf_fabric::resolve_machine_id` derives a stable,
unique identity from the host — `/etc/machine-id`, else a UUID generated once and
**persisted under the data dir**, else the hostname — so two nodes never collide
on the old `node-local` placeholder. The identity (keypair / Raft id / membership
id) keys off this, and a separate **friendly name** (`--node-name`/
`OCF_NODE_NAME`, default the hostname) is purely cosmetic. `GET /fabric/wireguard`
now reports both `node` (identity) and `name`. Verified live: a fresh node
auto-took its hostname, and a data-dir node generated + persisted a UUID stable
across restarts.

## 7. RBAC enforcement — key functionality behind authz

The RBAC engine existed but **nothing enforced it** — every endpoint was open.
Now an `authz` middleware (`ocf-api`) gates the API:

- It authenticates the caller from the `Authorization` header (HTTP Basic or a
  Bearer token) via the registered authenticators, then asks the `RbacEngine`
  whether that principal may perform the action (`AccessRequest`).
- `required_permission(method, path)` maps each request to a permission: writes
  under `/workloads` → `workload.manage`, `/networks` → `vpc.manage`,
  `/loadbalancers` → `lb.manage`, everything else that mutates (platform updates,
  health fixes, fabric machine fail/heartbeat) → `sys.modify`; `/access/*` and
  `/admin/*` are privileged for read *and* write; ordinary reads stay open so the
  dashboard works.
- Missing/invalid credentials → `401`; an authenticated-but-unpermitted principal
  → `403`. Default credentials `admin` / `admin`.

Verified live: an open read returns 200; a mutation returns 401 without
credentials, 401 with a wrong password, 200 with `admin:admin`; `GET
/access/users` is 401 without credentials and 200 with admin. Unit tests cover
the permission mapping, header parsing, and the authenticate-then-authorize path
(admin allowed, an unbound principal forbidden). Added `Error::unauthenticated`.

## Config / API

- New `fabric_raft_port` (default 51901). `--seed` / `OCF_SEEDS` (already present)
  now drives cluster join. New `--fabric-address` / `OCF_FABRIC_ADDRESS` (auto-
  detected when unset). `--node-id` is now optional (auto-derived); new
  `--node-name` / `OCF_NODE_NAME` and `--machine-id` / `OCF_MACHINE_ID`.

## Testing

- **265 tests pass.** New: the three fabric-cluster tests above. Both boot paths
  verified live — standalone (demo) and a seeds-configured joining node come up
